# Telegram `/tasks` 任务中心设计

日期：2026-07-10

## 目标

把 tg-vault Telegram Bot 的 `/tasks` 从混杂、偏命令行式的任务清单，改造成可选择任务并按任务执行开始、暂停、继续和取消的两级任务中心。

本次范围覆盖：

- 普通单文件下载：一次发送视为一个任务组；
- Telegram 相册：一个相册视为一个任务组；
- 频道日期/标签批量任务：一个后台 job 视为一个任务组；
- `/tasks` 列表页、详情页、取消确认页；
- 任务组级调度与控制；
- 旧命令兼容。

本次不做：

- 普通任务的数据库持久化与重启恢复；
- 单文件断点续传；
- 新的历史任务账本；
- 失败文件的细粒度选择重试 UI。

## 已确认的产品决策

1. 普通下载按“一次发送/一个相册”归组，不按每个文件单独展示。
2. 新任务保持自动开始，不增加手动审批步骤。
3. 运行中点击暂停时，当前文件继续完成；该组不再启动新文件，随后进入已暂停。
4. 等待任务的“开始”表示提升优先级，在下一个空闲下载位尽快开始，不抢占活动文件。
5. 采用现有下载队列增加任务组层的方案，不把所有普通任务改造成数据库任务。
6. 完成后构建、部署并验证 backend；未经明确要求不推送 GitHub。

## 用户交互

### 列表页

`/tasks` 回复一个可编辑的任务中心卡片：

```text
📥 下载任务

🟢 运行中 2　⏳ 等待 1　⏸ 已暂停 1
共 4 个进行中的任务

1. 🟢 夏日相册
   相册 · 4/12 · 当前：IMG_004.jpg

2. ⏳ report.pdf
   单文件 · 等待可用下载位

3. ⏸ @channel · #壁纸
   频道任务 · 38/120 · 已暂停
```

规则：

- 每页最多显示 6 个非终态任务；
- 每个任务对应一个编号选择按钮；
- 支持上一页、下一页和刷新；
- 不显示成功、失败、取消等历史任务；
- 普通任务和频道任务使用一致的状态术语和导航。

### 详情页

选择任务后编辑原消息，显示：

- 状态与暂停/系统阻塞原因；
- 类型、任务名称/来源；
- 文件统计及当前文件；
- 保存目标；
- 创建时间与最近更新时间；
- 状态对应的操作按钮。

操作按钮：

- 等待中：`▶️ 优先开始`；
- 运行中：`⏸ 完成本文件后暂停`；
- 暂停请求中：不再显示可重复暂停按钮；
- 已暂停：`▶️ 继续`；
- 所有非终态：`🛑 取消`；
- 固定导航：`↩️ 返回任务列表`、`🔄 刷新`。

### 取消确认页

取消必须经过确认页：

- 明确列出任务名称和影响范围；
- `⚠️ 确认取消` 执行任务级取消；
- `返回详情` 不改变任务状态；
- 成功取消后刷新列表；若任务中心已空，显示空状态并移除无效按钮。

## 任务组模型

新增轻量 `DownloadTaskGroup`：

```ts
type DownloadTaskGroupKind = 'single' | 'album';
type DownloadTaskGroupState =
  | 'waiting'
  | 'running'
  | 'pausing'
  | 'paused'
  | 'cancelling'
  | 'completed'
  | 'cancelled';

interface DownloadTaskGroup {
  id: string;
  kind: DownloadTaskGroupKind;
  title: string;
  chatId: string;
  userId?: number;
  source?: string;
  targetFolder?: string | null;
  state: DownloadTaskGroupState;
  createdAt: number;
  updatedAt: number;
}
```

每个底层文件任务增加 `groupId`。组聚合统计由队列内的 pending、active 与短期 group metadata 计算，避免重复维护容易漂移的计数。

频道任务仍以 `telegram_background_jobs` 和 `telegram_download_items` 为事实来源，通过适配器转换为相同的任务中心视图，不重复写入普通任务组存储。

## 调度与控制

### 自动开始

任务文件进入队列后正常自动调度，维持现有使用习惯。

### 优先开始

把选中组的等待文件稳定地移动到其它等待文件之前，保留组内原始顺序。不会中断已活动文件。

### 暂停

- 将组状态设为 `pausing`；
- 立即阻止该组新的等待文件被调度；
- 已活动文件继续执行；
- 该组最后一个活动文件结束后转为 `paused`；
- 如果点击时没有活动文件，则立即转为 `paused`。

### 继续

- 仅将选中组从 paused/pausing 恢复为 waiting；
- 重新触发调度；
- 不改变其它任务组，也不能绕过磁盘保护等系统级暂停。

### 取消

- 先把组状态锁为 cancelling，保证并发点击幂等；
- 从等待队列移除该组全部文件并结算 Promise；
- 对该组活动文件调用 AbortController；
- 活动文件退出后清理临时文件；
- 最终标记 cancelled 并从 `/tasks` 活跃列表移除；
- 不触碰其它组。

### 公平性

正常状态沿用入队顺序。用户对某组执行“优先开始”后，仅做一次稳定优先级提升；该组开始占用空闲位后，不永久垄断调度优先权。

## 频道任务适配

- 列表数据继续由 `listTelegramActiveTaskQueues` 提供；
- 详情页使用现有 item 聚合字段；
- 暂停、继续、取消分别调用现有 `pauseTelegramBackgroundJob`、`resumeTelegramBackgroundJob`、`cancelTelegramBackgroundJob`；
- 频道任务仍可跨 Bot 重启恢复；
- 终态、取消态和无实际活动的历史任务不进入 `/tasks`；
- 频道 job ID 的短前缀作为安全、紧凑的 callback selector。

## 安全与异常处理

- 所有回调重新校验用户认证；
- 普通任务按 chatId/userId 校验所有权；
- 频道任务按 user_id 查询并操作；
- 旧按钮、已完成任务返回“任务已结束或已失效”；
- callback data 使用短动作码和短 ID，保持在 Telegram 64 字节限制内；
- 页码在任务数变化后自动夹紧；
- 取消、暂停、继续均保持幂等；
- 磁盘压力、云盘限额和 FloodWait 等系统状态优先于用户继续操作；
- Telegram `MESSAGE_NOT_MODIFIED` 视为成功刷新，不弹错误警告。

## 兼容性

保留：

- `/task_pause <id>`；
- `/task_resume <id>`；
- `/task_cancel <id>`；
- 无 ID 的旧全局操作语义。

新的 `/tasks` 卡片不再显示需要用户手敲的控制命令。

## 测试策略

先把任务组调度器做成纯模块并按 TDD 覆盖：

1. 单文件和相册归组；
2. 自动开始；
3. 等待组稳定优先；
4. 暂停后不再启动新文件；
5. 活动文件完成后从 pausing 转 paused；
6. 继续只恢复选中组；
7. 取消只清理选中组并中断其活动文件；
8. 重复操作幂等；
9. 系统级暂停不可被组级继续绕过；
10. 列表分页、详情渲染、动态按钮和取消确认；
11. 所有权与过期按钮；
12. 频道任务适配及旧命令兼容。

## 交付验证

1. 运行新增单元测试与现有测试；
2. 运行 backend 构建；
3. 重建并强制重建 backend 容器；
4. 检查容器状态和 `/health`；
5. 检查 Telegram Bot 启动日志和错误日志；
6. 静态确认 callback 路由、按钮数据和任务级控制进入 `dist/index.js`；
7. 不推送远端，除非用户另行明确要求。
